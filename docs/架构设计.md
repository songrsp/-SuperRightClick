# 架构设计

## 一、整体结构

```
                         ┌─────────────────────────────┐
   用户右键点击           │     Finder (访达)            │
        │                └──────────────┬──────────────┘
        ▼                               │ 加载扩展
┌───────────────────────────┐          ▼
│  FinderExtension.appex     │   ← 沙盒，App 扩展
│  ┌──────────────────────┐  │
│  │ FinderSyncController  │  │   监控根目录 "/" → 任意位置都出菜单
│  │ MenuBuilder           │  │   构建仿 Windows 多级菜单
│  └──────────┬───────────┘  │
│             │ 写 JSON 指令  │
└─────────────┼──────────────┘
              ▼
   ┌──────────────────────┐
   │  App Group 共享信箱    │   group.com.team.superrightclick/mailbox/*.json
   └──────────┬───────────┘
              ▼ 轮询读取
┌───────────────────────────┐
│  SuperRightClick.app       │   ← 非沙盒，常驻后台
│  ┌──────────────────────┐  │
│  │ AppDelegate (0.5s轮询) │  │
│  │ CommandExecutor       │  │   执行文件操作 / shell / 哈希
│  │ UninstallManager      │  │   扫描残余 + 弹窗确认 + 移废纸篓
│  └──────────────────────┘  │
└───────────────────────────┘
```

## 二、为什么拆成两个进程

App 扩展（`.appex`）受 macOS 强制沙盒约束，无法：

- 删除 `~/Library`、`/Applications` 下的任意文件（卸载残余必须）
- 自由启动子进程跑 `defaults`、`ditto`、`killall Finder` 等

而把扩展关沙盒是不被允许的。于是采用业界通用做法：

- **扩展**：只做"轻 + 受限"的事 —— 出菜单、读取选中项、写剪贴板、投递指令。
- **主程序**：关闭沙盒（`com.apple.security.app-sandbox = false`），承担所有重操作。代价是只能 Developer ID 分发，不能上架商店。

## 三、进程间通信（IPC）

主通道：**App Group 共享目录信箱**。

1. 扩展把 `Command`（动作 + 路径列表）编码成 JSON，写入
   `containerURL(forSecurityApplicationGroupIdentifier:)/mailbox/<uuid>.json`
2. 扩展用 `NSWorkspace.openApplication` 把主程序唤起到后台
3. 主程序 `AppDelegate` 每 0.5s 轮询信箱，读到就执行，执行完删除该 JSON

> 为什么用"文件信箱 + 轮询"而不是 XPC？因为它**最稳、最好调试**：信箱里能直接看到积压指令，主程序没开也不会丢（下次启动补处理）。代价是有最多 0.5s 延迟，对右键操作完全够用。需要更低延迟可改用 FSEvents 监听信箱目录，或换 `NSXPCConnection`。

备用通道：自定义 URL Scheme `superrightclick://run?action=...&paths=<base64>`，由 `application(_:open:)` 接收。当 App Group 不可用时兜底。

## 四、菜单为什么能"到处出现"

```swift
FIFinderSyncController.default().directoryURLs = [URL(fileURLWithPath: "/")]
```

Finder Sync 扩展只对"被监控的目录"生效。把监控目录设为根目录 `/`，菜单就在任何文件夹都出现。若只想在指定目录生效（如 `~/Code`），把数组换成对应 URL 即可。

## 五、安全设计

- 卸载/清理一律 **`trashItem`（移到废纸篓）**，不做 `removeItem` 永久删除，给用户后悔余地。
- 残余扫描采用"白名单目录 + 关键词匹配"，只在已知的 `~/Library` 子目录里按 Bundle ID / 应用名匹配，避免全盘乱删。
- 删除前必有可视化确认窗口，逐项可勾选。

## 六、可扩展点

| 想做的事 | 改哪里 |
|----------|--------|
| 加一个右键功能 | `ActionID` + `MenuBuilder` + `CommandExecutor` |
| 改菜单顺序/分组 | `MenuBuilder.build` |
| 换默认编辑器 | `CommandExecutor.openVSCode` |
| 调残余扫描目录 | `UninstallManager.scanResiduals` 的 `searchDirs` |
| 降低指令延迟 | `AppDelegate` 改 FSEvents |
